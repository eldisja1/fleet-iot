# Database Schema

## Overview

Enterprise Fleet Tracking IoT Platform uses **PostgreSQL** to store device metadata and telemetry data generated by IoT fleet devices.

The database is designed to support:

- device registration
- telemetry ingestion from MQTT consumers
- efficient querying of historical device data
- scalable telemetry storage

The schema consists of two primary tables:

- `devices`
- `telemetry`

---

## Devices Table

The `devices` table stores metadata for registered IoT devices.

Each device represents a physical fleet unit capable of sending telemetry data.

| Column | Type | Description |
|------|------|------|
| id | UUID | Primary key |
| device_code | VARCHAR(100) | Unique device identifier |
| name | VARCHAR(255) | Optional device name |
| status | VARCHAR(50) | Current device status |
| created_at | TIMESTAMP | Device registration timestamp |
| updated_at | TIMESTAMP | Last update timestamp |

Example schema:

```
CREATE TABLE devices (
    id UUID PRIMARY KEY,
    device_code VARCHAR(100) UNIQUE NOT NULL,
    name VARCHAR(255),
    status VARCHAR(50),
    created_at TIMESTAMP,
    updated_at TIMESTAMP
);
```

---

## Telemetry Table

The `telemetry` table stores telemetry data generated by IoT devices.

Each telemetry record is associated with a device.

| Column | Type | Description |
|------|------|------|
| id | UUID | Primary key |
| device_id | UUID | Reference to device |
| latitude | DOUBLE PRECISION | Device latitude |
| longitude | DOUBLE PRECISION | Device longitude |
| speed | DOUBLE PRECISION | Vehicle speed |
| fuel_level | DOUBLE PRECISION | Fuel level percentage |
| status | VARCHAR(50) | Device or vehicle status |
| created_at | TIMESTAMP | Telemetry timestamp |

Example schema:

```
CREATE TABLE telemetry (
    id UUID PRIMARY KEY,
    device_id UUID NOT NULL,
    latitude DOUBLE PRECISION,
    longitude DOUBLE PRECISION,
    speed DOUBLE PRECISION,
    fuel_level DOUBLE PRECISION,
    status VARCHAR(50),
    created_at TIMESTAMP
);
```

## Table Relationships

The database uses a **one-to-many relationship** between devices and telemetry data.

![Entity Relationship Diagram](/docs/images/ER-diagram.png)

Relationship description:

- One device can produce multiple telemetry records.
- Each telemetry record belongs to exactly one device.

This relationship is enforced through a **foreign key constraint** between the `telemetry.device_id` column and the `devices.id` column.

---

## Indexes

Indexes are used to improve telemetry query performance.

Example index:

```
CREATE INDEX idx_telemetry_device_time
ON telemetry (device_id, created_at DESC);
```

This index optimizes queries such as:

- retrieving the latest telemetry for a device
- querying historical telemetry by device
- pagination of telemetry records
