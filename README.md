# Enterprise Fleet Tracking IoT Platform 

Fleet IoT Enterprise is a containerized telemetry ingestion platform designed to simulate and manage fleet vehicle data using MQTT messaging and a PostgreSQL database.

The system demonstrates a production-style IoT data pipeline where telemetry data from simulated devices is collected through an MQTT broker, processed by a consumer service, stored in a relational database, and exposed via a REST API.

This project is designed as a **learning and engineering portfolio project** that follows enterprise-style development practices including containerization, structured logging, monitoring readiness, and basic security hardening.

---

# System Overview

Fleet IoT Enterprise simulates a simplified real-world fleet tracking infrastructure.

Telemetry messages are generated by simulated devices and transmitted through an MQTT broker. A backend consumer processes these messages and stores them in a PostgreSQL database. A FastAPI service exposes REST endpoints for querying device and telemetry data.

All components run as isolated containers managed by Docker Compose.

## Telemetry Data Flow

The following steps describe how telemetry data flows through the system:

1. **Telemetry Generation**  
   The Device Simulator generates telemetry data representing fleet vehicle information such as location, speed, and device identifiers. This component emulates real-world IoT devices for development and testing.

2. **MQTT Message Publishing**  
   The simulator publishes telemetry messages to an MQTT topic using the MQTT protocol, which is optimized for lightweight communication in IoT environments.

3. **MQTT Broker Reception**  
   The MQTT Broker (Mosquitto) receives the message and distributes it to subscribed consumers, acting as a decoupling layer between devices and backend services.

4. **Message Validation**  
   The system verifies that the incoming MQTT payload is valid JSON. Invalid messages are discarded and logged to prevent malformed data from entering the pipeline.

5. **Consumer Processing**  
   The MQTT Consumer subscribes to the telemetry topic, retrieves the message, and parses the telemetry payload for further processing.

6. **Telemetry Schema Validation**  
   The parsed data is validated against the expected telemetry schema to ensure required fields and correct data formats.

7. **Data Transformation**  
   Valid telemetry data is prepared or normalized to match the database schema before storage.

8. **Database Insertion**  
   The processed telemetry record is inserted into the PostgreSQL database, which serves as the persistent storage layer.

9. **Database Error Handling**  
   If the database insertion fails, the system logs the error to support operational monitoring and troubleshooting.

10. **Data Access via API**  
   Once stored, the FastAPI backend retrieves telemetry records from PostgreSQL and exposes them through REST API endpoints.


# Core Components

## MQTT Broker

Mosquitto is used as the MQTT message broker responsible for receiving telemetry messages from IoT devices.

Features:

- topic-based messaging
- username/password authentication
- lightweight messaging protocol
- containerized deployment

---

## MQTT Consumer Service

The consumer service subscribes to MQTT topics and processes incoming telemetry data.

Responsibilities:

- subscribe to telemetry topics
- parse JSON payloads
- validate telemetry data
- insert records into PostgreSQL
- handle processing errors
- structured logging

Technologies used:

- Python
- Paho MQTT
- SQLAlchemy
- PostgreSQL driver

---

## PostgreSQL Database

PostgreSQL is used as the persistent storage for telemetry and device metadata.

Responsibilities:

- telemetry data storage
- relational data structure
- indexed queries for efficient access

---

## FastAPI Backend

FastAPI provides REST endpoints for accessing telemetry and device information.

Capabilities:

- device listing
- telemetry queries
- pagination support
- date range filtering
- authentication support

---

## Device Simulator

The simulator emulates multiple fleet devices sending telemetry data.

Features:

- multiple simulated vehicles
- GPS location updates
- speed simulation
- fuel level simulation
- periodic MQTT publishing

# Security Overview

The platform applies several baseline security practices.

### MQTT Security

- anonymous access disabled
- authenticated client connections

### API Security

- authentication required for API access

### Database Security

- database service not publicly exposed
- access restricted to internal containers

### Network Isolation

All services communicate through a private Docker network.

# Deployment

## Requirements

- Docker
- Docker Compose
- Git

---

## Clone Repository

```
git clone https://github.com/eldisja1/fleet-iot-enterprise.git
cd fleet-iot-enterprise
```

---

## Environment Configuration

Create an environment configuration file based on the example:

```
cp .env.example .env
```

Edit the `.env` file and provide appropriate values for your environment.

Sensitive credentials should **never be committed to the repository**.

---

## Start the Platform

```
docker-compose up -d
```

---

- mosquitto
- postgres
- consumer
- api
- simulator
